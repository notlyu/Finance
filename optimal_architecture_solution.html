Правила видимости и область применения scope

Scope определяет три вещи одновременно: кто видит запись, входит ли она в статистику, из какого «кошелька» берутся деньги.

Значения scope и их смысл

personal

Вижу только я. Не входит в семейную статистику, бюджеты, аналитику. Деньги — с моего личного счёта. Доступно только в личном пространстве.

family

Видят все участники семьи. Входит в семейную статистику. Деньги — с общего семейного счёта (scope='family' у Account). Только в семейном пространстве.

shared

Видят все участники. Входит в статистику с маркером «от [имя участника]». Деньги — с личного счёта участника, но операция видна семье. Используется когда: «я заплатил за семью со своей карты».

Правила пространства и автоматического scope

P
Личное пространство (/personal/*) — все записи scope='personal', family_id=NULL. Семейный переключатель недоступен.
F
Семейное пространство (/family/*) — по умолчанию scope='family'. Пользователь может выбрать 'shared' (я плачу за семью со своей карты). Scope 'personal' недоступен — личные операции добавляются только в личном пространстве.
A
Автоматические транзакции (recurring job) — наследуют scope от шаблона RecurringTransaction. Счёт берётся из account_id шаблона.

Подушка безопасности: какие счета считать

L
Личная подушка — все счета пользователя где is_liquid=true и scope='personal'.
F
Семейная подушка — только счета где is_liquid=true и scope='family' (family_id = текущая семья). Личные счета участников не учитываются.

Категории: правило видимости

S
Системные категории (is_system=true) — видны всем, нельзя удалить.
P
Личные категории (user_id!=NULL, family_id=NULL) — видны только создателю. В семейных dropdown не отображаются.
F
Семейные категории (family_id!=NULL) — видны всем участникам семьи.

Денежные потоки без дыр

Главная дыра старой архитектуры: транзакция существует, но баланс счёта не меняется (account_id необязательный). Решение — account_id обязателен везде, баланс обновляется в одном месте.

Механизм обновления баланса счёта

Единственное место изменения баланса — хук после записи Transaction в БД (Prisma middleware или database trigger):

  • Создание транзакции expense → Account.balance -= amount
  • Создание транзакции income → Account.balance += amount
  • Удаление транзакции → откат изменения баланса
  • Редактирование суммы → разница применяется к балансу

Атомарность Всё в одной транзакции БД. Либо и запись, и баланс меняются, либо ничего.

Recurring job: исправленный алгоритм

Текущая проблема: нет защиты от двойного запуска, нет счёта. Исправленный алгоритм:

  1. Выбрать все RecurringTransaction где active=true и last_run_month < текущий_месяц (или NULL)
  2. Для каждого — начать транзакцию БД: SELECT ... FOR UPDATE (блокировка строки)
  3. Создать Transaction с account_id из шаблона, scope из шаблона, date = day_of_month текущего месяца
  4. Если goal_id != NULL — создать GoalContribution, обновить Goal.current_amount
  5. Если debt_id != NULL — обновить Debt.remaining -= amount
  6. Обновить last_run_month = текущий_месяц
  7. Зафиксировать транзакцию. При ошибке — откат, запись в FailedJob

Долги: синхронизация remaining

Старая проблема: recurring платёж по долгу не уменьшал remaining. Ручное «закрыть часть» и автоматический платёж могли задвоиться.

Новое правило:

  • Любое погашение долга (ручное или автоматическое) создаёт Transaction + обновляет Debt.remaining в одной БД-транзакции
  • «Закрыть часть» — проверяет: если в этом месяце уже создан recurring-платёж по этому долгу, сумма учитывается. UI показывает уже погашено в этом месяце
  • При Debt.remaining <= 0 → автоматически is_active = false, уведомление пользователю, деактивация связанного RecurringTransaction

GoalContribution: защита от потери при удалении транзакции

Добавить ON DELETE SET NULL на GoalContribution.transaction_id, но при этом пересчитывать Goal.current_amount:

  • При удалении Transaction → GoalContribution.transaction_id = NULL, Goal.current_amount -= contribution.amount
  • Это реализуется через Prisma middleware: перехватить delete Transaction, найти связанные contributions, обновить цели
  • Аналогично для WishContribution

Результат Удаление транзакции всегда корректно отражается в прогрессе цели

auto_contribute_type = 'percent_income': как считать

Старая проблема: «% от дохода» — от чьего дохода в семейной цели?

Решение: процент считается от дохода создателя цели за предыдущий месяц. Алгоритм в interestJob (1-го числа):

  1. Найти все цели с auto_contribute_enabled=true и auto_contribute_type='percent_income'
  2. Для каждой: взять сумму income-транзакций user_id = goal.user_id за прошлый месяц
  3. Рассчитать сумму = total_income × (auto_contribute_value / 100)
  4. Создать RecurringTransaction с этой суммой (или сразу создать Transaction + Contribution)
